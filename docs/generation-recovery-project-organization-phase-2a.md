# Generation Recovery & Project Organization

## Phase 2A Implementation Report

Date: 2026-06-24

Scope: schema, migration, and backfill foundation only.

## Implemented

### New models

Added in `backend/models_new.py`:

1. `GenerationProject`
2. `GenerationRecord`
3. `GenerationRecoveryAudit`

Key behaviors implemented:

1. Generation projects are separate from task projects.
2. Generation records support nullable project assignment and nullable recovery audit linkage.
3. Generation records support nullable ownership for future unknown-owner flows.
4. Generation identity requires at least one of:
   - `provider_task_id`
   - `provider_generation_id`
   - `canonical_asset_key`

### New constraints and indexes

Implemented in models and mirrored in migrations:

1. Active project name uniqueness per owner:
   - `ux_generation_projects_owner_normalized_name_active`
2. Duplicate-prevention indexes for generation records:
   - `ux_generation_records_provider_task_id`
   - `ux_generation_records_provider_generation_id`
   - `ux_generation_records_provider_asset_key`
   - `ux_generation_records_source_usage_event_id`
3. Query indexes for owner/project/status/recovery access patterns.
4. Identity presence check constraint:
   - `ck_generation_records_identity_present`

### Migration implementation

Implemented inside the existing `db_migrations.py` pattern:

1. PostgreSQL path:
   - `_ensure_postgres_schema()`
2. SQLite compatibility path:
   - `ensure_operational_schema()` sqlite branch

Migration approach:

1. Keep `create_all()` as the primary table creator.
2. Keep `ensure_operational_schema()` as additive/idempotent safety net.
3. Use `CREATE TABLE IF NOT EXISTS`.
4. Use `CREATE INDEX IF NOT EXISTS`.
5. Add a PostgreSQL helper to avoid duplicate check-constraint creation.

### Backfill implementation

Added:

1. `backend/utils/generation_backfill.py`
2. `backend/backfill_generation_records.py`

Backfill capabilities:

1. Dry-run by default.
2. `--apply` mode for explicit writes.
3. Replay-safe duplicate prevention.
4. Batch scanning via `--batch-size`.
5. Resume support via `--start-after-id`.
6. Scan limiting via `--max-rows`.
7. Malformed-row tolerance.
8. Detailed summary output in JSON.

Backfill identity extraction currently uses:

1. Kling task-like identifiers from usage-event columns and metadata.
2. Generation identifiers from usage-event fields and generation-pipeline metadata.
3. Canonical output asset URL/key from `mediaAssets`.

Backfill deliberately does not:

1. modify task project fields
2. modify workspace UI
3. create API endpoints
4. assign generations to projects
5. import recovered external generations

## Backfill Summary

### Implemented behavior

The backfill script reads Kling usage events and attempts to create canonical `GenerationRecord` rows only when identity is strong enough to satisfy the schema.

Candidate evaluation outcomes:

1. non-generation rows skipped
2. generation-like rows with no usable identity skipped
3. duplicate source usage events skipped
4. duplicate identities skipped
5. valid canonical rows inserted or reported as would-insert in dry-run mode

### Pre-coding analysis result

Live database analysis could not be completed from this environment because even a basic PostgreSQL connectivity query timed out.

Fallback analysis from `supabase_final.sql` showed:

1. backup file date: 2026-05-21
2. usage-event rows in backup: `220`
3. Kling tool rows in backup: all `220` usage events were on tool id `4` (`kling`)
4. event type in backup: all `220` rows were `generate_click`
5. rows with prompt text in backup: `56`
6. rows with credits captured in backup: `140`
7. rows with metadata payloads in backup: `220`
8. rows with `mediaAssets` in backup sample analysis: `0`
9. rows with `klingTaskId` in backup sample analysis: `0`

Interpretation:

1. The backup suggests historical rows may be weakly identified.
2. The live schema's newer columns and richer metadata are likely critical for successful backfill.
3. The backfill implementation was therefore designed to be conservative, batchable, and dry-run-first.

## Risk Assessment

### Remaining risks

1. Live database reachability from this environment remains unresolved, so current row counts and true high-confidence backfill counts were not verified directly.
2. Historical legacy Kling rows may still lack enough identity to create `GenerationRecord` rows.
3. If the live table is much larger than the backup snapshot, backfill runtime may need tuning beyond the default batch size.
4. Existing `create_all()` plus additive migration flow remains fragile compared with a formal migration framework, though this change stayed within the established project pattern.

### Mitigations implemented

1. Dry-run is the default backfill mode.
2. Writes require explicit `--apply`.
3. Backfill is replay-safe and duplicate-aware.
4. Rows without identity are skipped instead of forcing weak inserts.
5. Malformed rows are logged and tolerated instead of aborting the entire run.
